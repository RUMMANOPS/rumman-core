# Catalog Patch Backlog (prioritized)

**التاريخ:** 2026-06-24
**المرجع:** `FULL_CATALOG_PLAN_CERTIFICATION_AUDIT.md` + `CATALOG_CERTIFICATION_DECISION_MATRIX.json`
**القاعدة:** P0 = يمنع تجربة أولى سيئة قبل App Store. P1 = مهم لكن لا يمنع الإطلاق إذا حُجبت/وُسمت الخطة. P2 = تحسينات لاحقة.

> **ملاحظة محورية:** بوجود الـ gate (حجب خطة HCI/PH/MHA/ENGT)، **لا شيء في P0 يمنع App Store** — لأن البرامج المكسورة لا تصل للطالب كخطة. ما يبقى P0 «حقيقي قبل المتجر» هو فقط ما قد يُسرّب خطأً للطالب رغم الـ gate (عناوين عربية للمقررات، الساعات، الـ gate نفسه).

## P0 — قبل App Store

### P0-1 — Mobile Plan Gate (الأهم)
- **المشكلة:** لا يوجد حاجز يمنع عرض خطة مكسورة.
- **المتأثر:** HCI, PH, MHA, ENGT (وأي برنامج مستقبلي غير معتمد).
- **أثر الطالب:** يرى ساعات/مستويات خاطئة من أول تجربة = فقدان ثقة.
- **يمنع App Store؟** **نعم** (هذا هو الحاجز نفسه).
- **يمنع plan map؟** هو من يحجبها.
- **يمنع Smart Registration؟** نعم لغير المعتمد.
- **نوع الإصلاح:** mobile rule (+ يُفضّل API field).
- **المخاطر:** منخفضة. لا يلمس بيانات.

### P0-2 — التحقق أن عناوين المقررات العربية لا تُعرض كنواقص/مشوّهة
- **المشكلة:** `official_title_ar` = NULL لـ100% من المقررات (من Audit سابق). PH/HCI/MHA مصادرها عربية بأكواد عربية (نجل001، صحة101).
- **المتأثر:** كل المقررات؛ حرج للبرامج العربية.
- **أثر الطالب:** أكواد/عناوين غير مفهومة لو ظهرت.
- **يمنع App Store؟** **نعم إذا كانت الخطة معروضة.** بما أن الخطط ذات المقررات العربية (PH/HCI) محجوبة أصلًا، **يسقط لـ P1** للبرامج المعروضة (كلها إنجليزية الأكواد). تبقى P0 فقط للتأكد أن الموبايل لا يُظهر كود مقرر بلا عنوان في البرامج المعروضة.
- **نوع الإصلاح:** mobile rule (fallback لعرض الكود بأناقة) الآن؛ data patch (عناوين عربية) لاحقًا.
- **المخاطر:** متوسطة.

### P0-3 — استخدام الساعات الرسمية لا مجموع الخطة (CS/IT/DS)
- **المشكلة:** مجموع الخطة (118/118/121) أقل من الرسمي (130/130/133) بسبب الاختياريات level=None.
- **المتأثر:** CS, IT, DS.
- **أثر الطالب:** يرى ساعات أقل من الحقيقة، وشريط تقدم خاطئ.
- **يمنع App Store؟** **نعم** لو عُرضت هذه البرامج (وهي معروضة بدلالات). الحل mobile rule (استخدم `official_total_credits`).
- **يمنع plan map؟** لا، يصحّحها.
- **نوع الإصلاح:** mobile rule.
- **المخاطر:** منخفضة.

### P0-4 — قاعدة «السنة الأولى المشتركة» بعلَم صريح (لا استنتاج من الفجوة)
- **المشكلة:** بدون علَم، الموبايل قد يستنتج السنة الأولى من «غياب level 2» — وهذا يُخفي خطأ HCI.
- **المتأثر:** MGT/ACC/FIN/ECOM/CS/IT/DS (تفعيل)، HCI (منع الاستنتاج الخاطئ).
- **أثر الطالب:** عرض «مستوى 1 ثم 3» مربك، أو إخفاء خطأ بيانات.
- **يمنع App Store؟** نعم للبرامج السبعة المعروضة.
- **نوع الإصلاح:** mobile rule الآن (قائمة برامج صريحة) + schema flag لاحقًا (`academic_phase`).
- **المخاطر:** منخفضة.

### P0-5 — Provenance fields (source_url/sha) — حد أدنى للحوكمة
- **المشكلة:** `source_document/url/sha256` = NULL لكل الـ19. لا أثر لمصدر أي خطة.
- **المتأثر:** كل البرامج.
- **أثر الطالب:** غير مباشر، لكنه يمنع التحقق/التدقيق ويُضعف الدفاع عند نزاع.
- **يمنع App Store؟** لا مباشرة. **يُنزَل لـ P1** ما لم يُطلب توثيق رسمي قبل الإطلاق.
- **نوع الإصلاح:** schema patch + data backfill.
- **المخاطر:** منخفضة.

- **نوع الإصلاح:** data patch (re-extract من المصدر العربي) + source acquisition (تأكيد المسار). **المخاطر:** متوسطة (إعادة استخراج عربي).

- **يمنع App Store؟** لا (محجوبة). **plan map؟** نعم. **Smart Reg؟** نعم.
- **نوع الإصلاح:** data patch (تفكيك السنوات لفصول) **أو** schema patch (`structure_type=year` + عرض سنوي). **المخاطر:** متوسطة-عالية (قرار نمذجة).
